## At a glance

```
GET    /api/health
POST   /api/runs                 # real repo run in the CloudboxRunner container
POST   /api/runs/:id/exec        # live run: follow-up command
GET    /api/runs/:id/read?path=...   # live run: read file
POST   /api/runs/:id/write       # live run: write file
POST   /api/runs/:id/dev         # live run: start/reuse dev process
POST   /api/runs/:id/stop        # live run: snapshot workspace to R2 and pause
POST   /api/runs/:id/resume      # live run: restore snapshot into the runner
POST   /api/runs/:id/fork        # live run: clone current workspace state
DELETE /api/runs/:id             # live run: delete active/snapshotted state
GET    /api/runs/:id/preview/*   # live run: secure preview proxy
POST   /api/computers            # materialize a ComputerSpec workspace
POST   /api/brief
GET    /api/c/:id/list
GET    /api/c/:id/read?path=...
POST   /api/c/:id/write
POST   /api/c/:id/ask
POST   /api/c/:id/submit
GET    /api/c/:id/grade
GET    /api/c/:id/receipts
GET    /api/c/:id/spec
```

When `CLOUDBOX_API_TOKEN` is set, authenticated endpoints require `Authorization: Bearer $CLOUDBOX_API_TOKEN` (or `x-cloudbox-token`).

## Health

```sh
curl http://localhost:8799/api/health
# → { "ok": true, "name": "cloudbox" }
```

## Run a real repo

The runner surface. Cloudbox boots the `cloudbox-runner` Cloudflare Container, clones a public GitHub repo, executes `commands`, then `verify`, collects the requested artifact, and returns the trail.

```sh
curl -X POST https://cloudbox.coey.dev/api/runs \
  -H "authorization: Bearer $CLOUDBOX_API_TOKEN" \
  -H 'content-type: application/json' \
  -d '{
    "repo": "https://github.com/acoyfellow/cloudbox",
    "commands": ["pnpm install --ignore-scripts"],
    "verify": ["pnpm run build", "pnpm run test"],
    "artifact": "HANDOFF.md"
  }'
```

Response shape:

```ts
type ContainerRunResult = {
  runId: string;
  ok: boolean;
  receipts: ContainerRunReceipt[];      // clone, command, verify, diff steps
  runnerReceipts: RunnerLifecycleEvent[]; // container boot / request / error
  artifact?: { path: string; content: string } | null;
  diff?: string;
  error?: string;
};
```

Each `ContainerRunReceipt` carries `cmd`, `code`, `signal`, `stdout`, `stderr`, `startedAt`, `finishedAt`. Non-zero verify exit codes set `ok: false` and return HTTP `422`.

`runnerReceipts` are the computer proof layer. They show whether the `CloudboxRunner` Durable Object started or reused a Container, how many readiness attempts were needed, how long boot took, and when the runner responded. Typical events are `runner.container.start`, `runner.container.ready_attempt`, `runner.container.ready`, `runner.response`, and `runner.container.not_ready`.

Recent runs are available when D1 is bound:

```sh
GET /api/runs/recent
GET /api/runs/:runId
```

## Keep a run alive

Set `"live": true` when creating a run to retain the cloned repo in the runner long enough for human or agent steering.

```sh
curl -X POST https://cloudbox.coey.dev/api/runs \
  -H "authorization: Bearer $CLOUDBOX_API_TOKEN" \
  -H 'content-type: application/json' \
  -d '{
    "repo": "https://github.com/acoyfellow/cloudbox",
    "verify": ["test -f package.json"],
    "live": true
  }'
```

Then use the returned `runId`:

```sh
curl -X POST https://cloudbox.coey.dev/api/runs/$RUN_ID/exec \
  -H "authorization: Bearer $CLOUDBOX_API_TOKEN" \
  -H 'content-type: application/json' \
  -d '{"command":"pwd"}'

curl "https://cloudbox.coey.dev/api/runs/$RUN_ID/read?path=README.md" \
  -H "authorization: Bearer $CLOUDBOX_API_TOKEN"

curl -X POST https://cloudbox.coey.dev/api/runs/$RUN_ID/write \
  -H "authorization: Bearer $CLOUDBOX_API_TOKEN" \
  -H 'content-type: application/json' \
  -d '{"path":"notes/live.md","content":"hello"}'
```

## Start a dev preview

A live run can launch a long-running process and proxy traffic through Cloudbox. The proxy preserves HTTP and WebSocket traffic so dev-server preview loops can sit behind one authenticated URL.

```sh
curl -X POST https://cloudbox.coey.dev/api/runs/$RUN_ID/dev \
  -H "authorization: Bearer $CLOUDBOX_API_TOKEN" \
  -H 'content-type: application/json' \
  -d '{"command":"bun run dev --host 0.0.0.0","port":5173}'

open "https://cloudbox.coey.dev/api/runs/$RUN_ID/preview/"
```

For a quick session helper:

```sh
CLOUDBOX_API_TOKEN=... bun run live:preview
# → { "runId": "...", "previewUrl": "..." }
```

For a steerable session CLI:

```sh
CLOUDBOX_API_TOKEN=... bun run live start \
  https://github.com/acoyfellow/cloudbox \
  "bun run dev --host 0.0.0.0" \
  5173

bun run live open
bun run live exec "pwd"
bun run live read README.md
bun run live write notes/live.md "agent changed this"
```

`start` writes `.cloudbox-live.json` in the current directory. The one-way sync helper reads that state and pushes local text-file edits into the live run:

```sh
CLOUDBOX_API_TOKEN=... bun run live:sync
```

Programmatic agent/MCP adapters can build on `cloudbox/live-run-tools`, which exposes the same create/dev/read/write/exec/preview/stop/resume/fork/delete primitives without introducing a second public product surface.

## Stop, resume, and fork a live workspace

Live runs accept an optional `ttlSeconds` value between 60 seconds and 30 days. The default lifetime is one hour. Stopping archives the repo workspace into the configured R2 artifacts bucket; resuming restores it into a live runner. Forking snapshots the source and restores the same workspace into an independent child run.

```sh
curl -X POST "https://cloudbox.coey.dev/api/runs/$RUN_ID/stop" \
  -H "authorization: Bearer $CLOUDBOX_API_TOKEN"

curl -X POST "https://cloudbox.coey.dev/api/runs/$RUN_ID/resume" \
  -H "authorization: Bearer $CLOUDBOX_API_TOKEN"

curl -X POST "https://cloudbox.coey.dev/api/runs/$RUN_ID/fork" \
  -H "authorization: Bearer $CLOUDBOX_API_TOKEN"
```

The session helper includes matching lifecycle commands:

```sh
bun run live info
bun run live stop
bun run live resume
bun run live fork
bun run live delete
```

## Browser shell and desktop prototype

Set `"desktop": true` together with `"live": true` to route the run to the heavier desktop runner binding rather than the normal proof runner. The Debian prototype serves a browser terminal through shellinabox and a lightweight Xvfb/noVNC Chromium desktop through the authenticated preview proxy:

```sh
bun run live start https://github.com/acoyfellow/cloudbox "bun run dev --host 0.0.0.0" 5173 --desktop
bun run live shell
bun run live desktop
```

Cloudflare Containers' account/operator SSH is not the hosted end-user interface here; shell access is intentionally carried over authenticated HTTP/WebSocket preview routes.

For a deployed smoke test, run:

```sh
CLOUDBOX_E2E_URL=https://cloudbox.coey.dev \
CLOUDBOX_API_TOKEN=... \
bun run proof:live-preview
```

## Materialize a workspace

```sh
curl -X POST http://localhost:8799/api/computers \
  -H 'content-type: application/json' \
  -d @spec.json
# → { "id": "cb_abc123…", "baseUrl": "/api/c/cb_abc123…" }
```

The `id` is a deterministic content hash of the spec. The same spec returns the same id every time — re-materializing is idempotent.

## Brief → spec

```sh
curl -X POST http://localhost:8799/api/brief \
  -H 'content-type: application/json' \
  -d '{"brief":"A staff engineer triaging an open PR with failing CI."}'
# → { "spec": { profile, filesystem, collaborators, objectives, rubric } }
```

The draft spec is a starting point — edit it, then POST it to `/api/computers` to materialize. v0 is a deterministic stub; the Workers AI-backed structured generator lands in a later phase.

## List files

```sh
curl http://localhost:8799/api/c/$ID/list
# → { "files": [{ "path": "...", "kind": "...", "state": "..." }, ...] }
```

## Read a file

```sh
curl "http://localhost:8799/api/c/$ID/read?path=docs/auth-redesign.md"
# → { "path": "...", "kind": "...", "content": "..." }
```

Records a `read` receipt.

## Write a file

```sh
curl -X POST http://localhost:8799/api/c/$ID/write \
  -H 'content-type: application/json' \
  -d '{"path":"notes/triage.md","content":"# Notes\n…"}'
# → { "path": "...", "written": <bytes> }
```

Records a `write` receipt.

## Ask a collaborator

```sh
curl -X POST http://localhost:8799/api/c/$ID/ask \
  -H 'content-type: application/json' \
  -d '{"who":"arch","message":"Should this use a queue?"}'
# → { "from": "arch", "role": "reviewer", "style": "architectural", "reply": "..." }
```

Records an `ask` receipt. The `who` value is the collaborator's `id` from the spec.

## Submit a decision

```sh
curl -X POST http://localhost:8799/api/c/$ID/submit \
  -H 'content-type: application/json' \
  -d '{"objective":"triage","decision":"request-changes","notes":"..."}'
# → { "objective": "...", "accepted": true }
```

Records a `submit` receipt. `objective` is the objective's `id` from the spec.

## Grade

```sh
curl http://localhost:8799/api/c/$ID/grade
# → {
#     "score": 4,
#     "max": 5,
#     "totalWeight": 6,
#     "passed": ["design-first", "decided"],
#     "failed": ["right-reviewer"],
#     "ungraded": ["explain-ci"],
#     "detail": [...]
#   }
```

Pure replay of the receipt log against the rubric. Idempotent — call it as many times as you want; the score is a function of the receipts at that moment.

## Receipts

```sh
curl http://localhost:8799/api/c/$ID/receipts
# → { "receipts": [{ "ts", "kind", "payload" }, ...] }
```

The full append-only log. Useful for debugging an agent's behavior.

## Spec

```sh
curl http://localhost:8799/api/c/$ID/spec
# → the original ComputerSpec
```
